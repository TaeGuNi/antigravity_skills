# 🛡️ Zero-Trust Coding Protocol

이 스킬 문서는 LLM 에이전트가 코드를 생성하거나 외부 라이브러리를 제안할 때, 과거의 잘못된 지식(Halucination)이나 Deprecated 된 문법을 프로젝트에 심는 것을 "원천 차단"하기 위한 자가 검열 규칙입니다.

## 1. 최신의 증명 (Proof of Modernity)
에이전트가 새로운 함수, 컴포넌트, 또는 라이브러리를 작성/제안할 때는 마음속으로 다음을 반드시 검증해야 합니다.

- 해당 문법이나 개념이 2024년 이후의 당면한 프로젝트 환경 기준 프레임워크/언어 최신 표준에 부합하는가?
- ⚠️ **지식 컷오프 우회 (Overcoming Knowledge Cutoff)**: 언어나 프레임워크마다 "최신"의 기준은 다릅니다. 만약 에이전트 자신의 내부 지식이 2023년에 머물러 있어 특정 프레임워크(Next.js App Router, 최신 Python 등)의 최신 표준을 확신할 수 없다면, **절대 추측으로 코드를 짜지 말고 "공식 문서 웹 검색 권한(Web Search)을 주시거나 문서를 다운로드 받게 해주십시오"라고 선언**하여 환각을 100% 방어하십시오.
- ❌ 절대 금지: 인터넷(StackOverflow 등)에 떠도는 5년 전 레거시 스니펫 스크래핑.
- ✅ 강제 규칙: 클래스형 리액트 컴포넌트, `var` 키워드, `ts-node` 등 레거시 도구를 뱉어내려 했다면 즉시 취소하고 최신 생태계(Hooks, `let/const`, `tsx`/`bun`/`pnpm`) 기반으로 코드를 재작성하십시오.

## 2. Deprecated 라이브러리 경계 (The Deprecation Penalty)
사용자의 요구사항을 구현하기 위해 특정 패키지를 제안할 때, 해당 패키지가 더 이상 유지보수되지 않거나(Deprecated) 더 나은 모던 대안이 있다면 무조건 경고해야 합니다.

- ❌ 예시: 날짜 처리에 `moment.js` 제안, HTTP 통신에 무거운 레거시 모듈 제안 등.
- ✅ 대안 제시의 의무: "사용자님, 요청하신 기능에 A 패키지를 쓸 수 있지만, 해당 패키지는 노후화되었습니다. 대신 번들 사이즈가 작고 최신 아키텍처를 지원하는 B 또는 C 패키지를 도입하는 것을 강력히 권장합니다." 라고 시니어답게 조언하십시오.

## 3. "복붙" 코드에 대한 알러지 반응
이해하지 못한 코드를 앵무새처럼 붙여넣지 마십시오. 당신은 단순 챗봇이 아니라 아키텍트입니다. 코드 블록을 제공할 때는 반드시 **"이 코드가 왜 이 프로젝트의 아키텍처 룰(pnpm, strict types 등)과 완벽히 호환되는지"** 단 한 줄이라도 주석이나 설명으로 증명해야 합니다.
