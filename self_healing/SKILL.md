# 🧟‍♂️ Self-Healing Protocol (에러 자가 치유 헌법)

이 스킬 문서는 에이전트가 코드를 작성하고 터미널에서 스크립트(컴파일, 린팅, 테스트 등)를 실행했을 때 빨간 줄(Error)이 발생하더라도, 나약하게 사용자에게 "어떻게 할까요?"라고 묻는 행위를 엄단하는 **"무한 재도전(Zombie) 아키텍처"**입니다. 인간에게 코딩을 완벽히 위임(Delegation) 받기 위한 가장 중요한 규율입니다.

## 1. 사용자에 대한 핑계 금지 (No Excuses, Do Not Ask)
명령어 실행 후 에러가 났을 때 즉시 `notify_user`나 채팅으로 결정을 떠넘기는 것은 에이전트의 직무 유기입니다.

- ❌ 금지 사례: "타입 에러가 발생했습니다. 이대로 진행할까요, 아니면 제가 고쳐볼까요?"
- ✅ 강제 원칙: 컴파일 에러, 린트 에러, 테스트 실패가 발생하면 **절대 사용자를 호출(Notify)하지 마십시오.** 스스로 디버깅 루프에 진입해야 합니다.

## 2. 로그 기반 자기 분석 (Trace & Understand)
에러 덩어리를 무지성으로 검색하지 말고 핀포인트를 찾아야 합니다.

- ✅ 강제 원칙: 터미널 출력 결과가 길다면, 스스로 에러 로그(Stack Trace)의 **최상단(Top) 30줄**을 읽고 에러의 근본 원인을 철저히 분석하십시오. 파일 경로와 라인 번호가 주어졌다면 그곳을 즉시 `view_file`로 까보아야 합니다.

## 3. 수정-실행의 자율 루프 (The Healing Loop)
원인을 파악했다면 추측으로 끝내지 말고, 증명해야 합니다.

- ✅ 강제 원칙: 에러를 유발한 코드를 수정한 뒤, 터미널에서 **다시 실행(Re-run) 명령어를 날리십시오.**
- 🔁 이 수정-실행-실패 파악 루프를 **최소 3번 반복**하십시오.
- 🟢 마침내 에러가 사라지고 녹색 점등(Green pass / Exit code 0)을 본 후에만 사용자에게 "발생한 N개의 에러를 제 스스로 분석하여 완벽하게 해결(Self-Healed)했습니다"라고 건조하게 보고하십시오.
